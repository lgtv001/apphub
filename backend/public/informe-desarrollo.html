Por qué tantas iteraciones en el árbol jerárquico Cada nivel (Área → Subárea → Sistema → Subsistema) tiene su propio estado de visibilidad, select en cascada, validación de padre y mini-barra de avance. Cada una de esas funciones interactuaba con las demás, generando bugs de "reset de formulario que no limpia el estado anterior". Fue el componente con mayor densidad lógica de todo el frontend.
3. Stack tecnológico y justificación
TecnologíaRol en el proyectoPor qué se eligió
Laravel 12 (PHP 8.2)
Backend		 API REST, ORM, migraciones, middleware, tests Ecosistema maduro para proyectos de gestión. Sanctum, Socialite y Eloquent reducen código boilerplate. Amplia comunidad en Latinoamérica facilita mantención futura.
Laravel Sanctum
Auth		 Tokens de API para SPA + protección de rutas Más ligero que Passport. Perfecto para SPA que no necesitan OAuth completo propio.
Laravel Socialite
OAuth		 Autenticación con GitHub y Google Abstrae el protocolo OAuth 2.0 completo en 2 líneas de código. Evita implementar manualmente intercambio de tokens, state CSRF, refresh tokens.
MySQL
Base de datos		 Persistencia de todos los datos del proyecto Modelo relacional natural para la jerarquía área→subsistema. Soporte nativo en Railway y la mayoría de hostings.
HTML + CSS + Vanilla JS (ES Modules)
Frontend		 Todas las páginas de la app Sin framework = sin bundle, sin npm en producción, carga instantánea. Para una app interna con equipos no-técnicos es más fácil de mantener que React/Vue. El diseño system propio en CSS variables permite modo claro/oscuro sin dependencias.
PHPUnit (via Artisan)
Tests		 85 tests de integración sobre base de datos real Se eligió integración sobre mocks porque en proyectos anteriores los mocks ocultaron bugs de migración. Cada endpoint se prueba contra una BD SQLite en memoria.
Railway.app
Hosting		 Deploy automático desde GitHub Soporta PHP 8.2, MySQL gestionado, variables de entorno por UI, deploy en push. Sin gestión de servidor.
Maatwebsite Excel
Importación		 Lectura de archivos .xlsx para importación masiva El único paquete Laravel con soporte robusto para Excel sin depender de LibreOffice.
4. Decisiones de arquitectura importantes
DecisiónAlternativa descartadaImpacto
SPA con HTML estático + API separada Blade templates (Laravel full-stack) El frontend puede ser servido desde CDN o cualquier servidor estático. El backend es intercambiable. Permite escalar ambos de forma independiente.
Rutas OAuth en web.php (no API) Rutas API sin sesión Socialite requiere sesión PHP para el parámetro state CSRF. Mezclar con API sin sesión rompe el flujo de seguridad OAuth.
solicitudes_acceso separado de usuarios Crear usuario inactivo al registrarse Permite que el superuser vea la solicitud antes de que el usuario exista en el sistema. Evita cuentas zombie si la solicitud es rechazada.
Logs de auditoría por tabla (no log global) Una tabla de log unificada Cada tabla de log tiene la estructura exacta de su entidad. Consultas más rápidas. Facilita purga de logs antiguos sin afectar otros módulos.
Middleware check.project en API Validar proyecto en cada controller Un solo punto de control garantiza que ningún usuario acceda a datos de un proyecto al que no pertenece, sin repetir la lógica en 20+ métodos.
Tests de integración sobre BD real (SQLite :memory:) Mocks de repositorios Los tests detectan errores de migración, constraints de clave foránea y lógica de Eloquent que los mocks no capturarían.
CSS variables + prefers-color-scheme Clase .dark-mode en body El tema sigue automáticamente la preferencia del sistema operativo del usuario. Cero JavaScript. Funciona en todos los navegadores modernos.

6. Compatibilidad con Bluehosting u otros hostings compartidos

Alternativas recomendadas si se quiere salir de Railway 1. DigitalOcean App Platform — misma experiencia que Railway, soporte nativo Laravel, deploy desde GitHub.
2. Render.com — plan gratuito disponible, soporta PHP/MySQL, deploy automático.
3. Bluehosting VPS — requiere configuración inicial (Nginx + PHP-FPM + MySQL) pero una vez configurado es completamente funcional y económico para uso en Chile.

En todos los casos se necesitan las mismas variables de entorno (DB_*, APP_KEY, GITHUB_*, GOOGLE_*) y ejecutar php artisan migrate --force en el primer deploy.
7. Resumen ejecutivo — ¿Por qué tomó tantas consultas?
FactorExplicación
Dominio complejo La jerarquía de 4 niveles con validaciones cruzadas (un código puede repetirse entre proyectos pero no dentro del mismo) requirió múltiples rondas de refinamiento en validaciones backend y frontend.
SPA sin framework Sin React/Vue, todo el manejo de estado (selects en cascada, modales de edición, reset de formularios) fue implementado manualmente. Más control, pero más superficie para bugs de estado.
Dos capas independientes Backend Laravel (API) y Frontend HTML/JS son capas desacopladas. Cada cambio en la API requería verificar que el JS lo consumiera correctamente, y viceversa.
Calidad con tests Se escribieron 85 tests de integración. Cada feature requirió escribir primero el test, ejecutarlo en rojo, implementar, ejecutarlo en verde. Más iteraciones, pero 0 regresiones durante todo el desarrollo.
Flujo OAuth desde cero Implementar un flujo "solicitar acceso" en lugar de "login directo" no tiene una receta estándar. Requirió diseñar la tabla, el flujo de redirección web, los estados y la interfaz de aprobación del superuser.
Decisiones de UX iterativas El diseño del dashboard, los iconos, el modo claro/oscuro y las tarjetas del hub fueron refinados según feedback directo durante el desarrollo. Cada ajuste visual fue una consulta adicional.
Conclusión La cantidad de consultas refleja el nivel de completitud de la aplicación: no se construyó un prototipo, sino una plataforma funcional con autenticación, autorización por roles, auditoría, importación de datos, dashboard, OAuth y panel de administración. En un equipo de desarrollo tradicional, esto equivale a 3–4 semanas de trabajo de un desarrollador full-stack senior.
